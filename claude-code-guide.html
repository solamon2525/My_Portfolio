Global CLAUDE.md

~/.claude/CLAUDE.md — ใช้กับทุก project เก็บ preferences ส่วนตัว เช่น ภาษา code style

📁

Project CLAUDE.md

ที่ root ของ project — tech stack, architecture, build commands, conventions เฉพาะ project

CLAUDE.md ตัวอย่าง # Project: E-Commerce API ## Tech Stack - Node.js 20 + Express 5 + TypeScript - PostgreSQL 16 (Drizzle ORM) + Redis ## คำสั่งสำคัญ - Build: `bun run build` # ใช้ bun ไม่ใช่ npm! - Test: `bun test` - Dev: `bun dev` ## กฎสำคัญ (Imperative) - ใช้ async/await เสมอ ห้ามใช้ callback - Error ทุกตัวต้อง log ด้วย logger.error() - ห้ามแก้ไขไฟล์ใน /src/generated/ (auto-generated) ## โครงสร้าง - /src/routes/ — API endpoints - /src/services/ — Business logic - /src/models/ — DB schemas
✅
Imperative Style ได้ผล 94% — เขียนแบบ "ใช้ async/await เสมอ" ดีกว่า "ควรจะใช้" compliance rate ต่างกัน 21%
⚠️
ความยาวไม่เกิน 200 บรรทัด — CLAUDE.md ที่ยาวเกินทำให้ใช้ token มากขึ้นและตอบแย่ลง รัน /init เพื่อสร้างอัตโนมัติ
🚫 .claudeignore — ตัดไฟล์ที่ไม่จำเป็น
.claudeignore # Build outputs dist/ build/ .next/ # Dependencies node_modules/ vendor/ # Generated + Lock files src/generated/ package-lock.json bun.lockb # Logs *.log coverage/
✅
ลด response time ได้ถึง 60% — โปรเจค 1,000 ไฟล์ที่ใช้ .claudeignore ที่ดีตอบเร็วจาก 8 วินาที เหลือ 3 วินาที

🔨 Custom Commands — สร้างคำสั่งเอง

สร้างไฟล์ .md ใน .claude/commands/ — ใช้เป็น slash command ได้ทันที ใช้ $ARGUMENTS รับ parameter

.claude/commands/review-pr.md ทำการ Code Review สำหรับ PR #$ARGUMENTS 1. รัน `gh pr diff $ARGUMENTS` 2. ตรวจหา: logic errors, security issues, missing error handling 3. สรุปเป็น: - 🔴 Critical: ต้องแก้ก่อน merge - 🟡 Warning: ควรแก้ถ้ามีเวลา - 🟢 Suggestion: ปรับปรุงได้ # ใช้งาน: /review-pr 123

🔌 Connectors / MCP

Model Context Protocol — มาตรฐานเชื่อมต่อ AI กับ external tools และ services ทุกชนิด

🧩 MCP คืออะไร

MCP เปรียบเหมือน "USB-C for AI" — มาตรฐาน open protocol ของ Anthropic ที่ทำให้ Claude เชื่อมต่อกับ external systems ได้อย่างปลอดภัยและมีโครงสร้าง

🔒

Secure

Auth ผ่าน browser session — Claude ไม่เห็น password ของคุณ

🔌

Standardized

Protocol เดียวกันทุก connector — เรียนรู้ครั้งเดียวใช้ได้ทุกที่

⚡

Extensible

สร้าง custom MCP server เองได้ เชื่อมกับ internal systems

➕ วิธีเพิ่ม Connector
# รูปแบบทั่วไป claude mcp add --transport sse [ชื่อ] [MCP server URL] # ตัวอย่าง claude mcp add --transport sse asana https://mcp.asana.com/sse # ดู connections ทั้งหมด + auth /mcp

👨‍💻 1. Software Development
Feature Development

สร้าง CRUD API ครบชุดใน 3 นาที

"สร้าง REST API สำหรับ User management มี CRUD ครบ, pagination, JWT auth, unit tests ใช้ Express + TypeScript + Drizzle ORM"
  • Claude สร้างไฟล์ครบชุด: routes, controllers, services, models, tests
  • มี error handling, validation, และ OpenAPI spec
  • เขียน tests ให้ 80%+ coverage อัตโนมัติ
Debugging

ตามหา Bug ข้ามไฟล์

"มี bug ที่ทำให้ user logout อัตโนมัติทุก 5 นาที แม้ตั้ง session timeout ไว้ 1 ชั่วโมง ช่วยหาสาเหตุ"
  • Claude อ่าน auth middleware, session config, token refresh logic
  • trace data flow พบว่า refresh token endpoint ส่ง wrong expiry
  • แก้ไขพร้อมเพิ่ม test ป้องกัน regression

Role Assignment

ให้ Claude รับบทเป็น Expert ในด้านที่ต้องการ

Beginner High Impact

การบอกให้ Claude รับ "บทบาท" ทำให้คำตอบมี depth และ context ที่ตรงกับงานมากขึ้น — เหมือนจ้าง specialist แทน generalist

🏗️

Template

คุณคือ [role] ที่เชี่ยวชาญ [domain]
ช่วยฉัน [task]

🎯

เหมาะกับ

Code review, Architecture design, Security audit, Performance tuning

📍

วาง Role ได้ที่

ใส่ใน CLAUDE.md (ถาวร) หรือพิมพ์ในแต่ละ session

❌ ไม่มี Role
# prompt ทั่วไป ช่วยดู code นี้ให้หน่อย # ผลลัพธ์: คำตอบผิวเผิน # ไม่ได้ focus ประเด็นสำคัญ
✅ มี Role
# prompt พร้อม role คุณคือ senior TypeScript engineer ที่เชี่ยวชาญ Node.js และ security ช่วย review code นี้ โดยเน้น: - SQL injection risks - Input validation - Error handling patterns # ผลลัพธ์: ลึก ครบ ตรงจุด
🏆 Pro Tip: วาง role ใน CLAUDE.md ไม่ต้องพิมพ์ซ้ำทุก session
# CLAUDE.md ## Role คุณคือ senior full-stack engineer ที่เชี่ยวชาญ React, TypeScript, Node.js และ PostgreSQL เน้น clean code, SOLID principles, และ security best practices

Chain of Thought

ให้ Claude คิดก่อนทำ — วิเคราะห์ทีละขั้น

Intermediate Complex Tasks

แทนที่จะให้ Claude ตอบทันที บอกให้ "คิดก่อน" — ผลลัพธ์จะถูกต้องมากขึ้น โดยเฉพาะงานซับซ้อน

🧠

เหมาะกับ

Complex architecture, Debugging tricky bugs, Refactoring decisions, Security analysis

⚡

คู่กับ Plan Mode

ใช้ Shift+Tab เปิด Plan Mode + Chain of Thought = ผลลัพธ์ดีที่สุด

🔑

คำสำคัญ

"วิเคราะห์ก่อน", "คิดทีละขั้น", "ข้อดี-ข้อเสีย", "เหตุผล"

❌ ไม่ใช้ CoT
migrate database schema จาก MongoDB ไป PostgreSQL ให้หน่อย # Claude เดาแล้วลุย # อาจพลาด edge cases สำคัญ
✅ ใช้ Chain of Thought
ก่อน migrate ขอให้: 1. วิเคราะห์ schema ปัจจุบัน 2. ระบุ data types ที่ต้องแปลง 3. หา relationships ที่ซับซ้อน 4. ชี้ risks และ data loss risks 5. เสนอ migration strategy แล้วค่อย implement migration script
💡 Trigger phrases สำหรับ Chain of Thought:
"คิดทีละขั้น", "วิเคราะห์ก่อน implement", "อธิบาย reasoning", "pros and cons ก่อน", "step by step"
🎯

Few-shot Prompting

ให้ตัวอย่าง 2-3 อัน แล้วค่อยถามงานจริง

Intermediate Style Matching

Claude เรียนรู้ pattern จากตัวอย่างที่คุณให้ได้ในทันที ใช้เพื่อบังคับ code style, naming convention, หรือ output format ที่ต้องการ

🧪

เหมาะกับ

เขียน tests, สร้าง boilerplate, naming conventions, commit message style

📏

จำนวนตัวอย่าง

2-3 ตัวอย่างเพียงพอ — มากเกินไปเปลือง token ไม่ได้ผลดีขึ้น

🎨

ผลลัพธ์

Output ที่ match style โปรเจคปัจจุบัน ไม่ต้อง edit เพิ่ม

❌ ไม่มีตัวอย่าง
เขียน unit test สำหรับ UserService.createUser()
✅ Few-shot
# Test patterns ในโปรเจคนี้: describe('OrderService', () => { it('should throw if stock < 0', async () => { await expect(svc.create({qty:-1})) .rejects.toThrow('INVALID_STOCK') }) }) # ตอนนี้เขียน test สำหรับ # UserService.createUser() ในรูปแบบเดียวกัน
ℹ️ Few-shot ใน CLAUDE.md: วาง example patterns ใน CLAUDE.md ได้เลย จะ apply กับทุก session โดยอัตโนมัติ

Constraint Prompting

ระบุขอบเขตชัดเจน — ห้ามทำ / ต้องทำ / ขนาดที่ต้องการ

Beginner ลด Error 40%+

Prompt ที่ไม่มีขอบเขตทำให้ Claude ตัดสินใจเองในสิ่งที่อาจไม่ต้องการ การระบุ constraint ชัดเจนช่วยลด unexpected changes

🚫

ห้ามทำ

ห้ามเปลี่ยน interface, ห้าม refactor ส่วนอื่น, ห้ามเพิ่ม dependency ใหม่

✅

ต้องทำ

ต้องผ่าน existing tests, ต้องใช้ TypeScript strict mode, ต้อง backward compatible

📏

ขนาด / Scope

แก้เฉพาะ function นี้, ไม่เกิน 50 lines, อธิบายสั้นไม่เกิน 3 bullets

❌ ไม่มี Constraint
ช่วย optimize ฟังก์ชัน getUserById() # Claude อาจ: # - เปลี่ยน function signature # - Refactor ทั้ง file # - เพิ่ม library ใหม่
✅ มี Constraint
ช่วย optimize getUserById(): ✅ แก้เฉพาะ function นี้เท่านั้น ✅ ต้องผ่าน existing unit tests ✅ ต้อง type-safe (TypeScript strict) 🚫 ห้ามเปลี่ยน function signature 🚫 ห้ามเพิ่ม external dependencies # ผลลัพธ์: targeted, safe, predictable
⚠️ Claude ใจดีเกินไป: Claude มักพยายาม "ช่วยเพิ่ม" โดยไม่ได้ถูกขอ — ระบุ constraint ป้องกันการแตะ code ที่ไม่ต้องการ
🔄

Iterative Prompting

พัฒนาทีละรอบ — draft แล้วค่อยๆ refine

Intermediate ประหยัด Token

แทนที่จะใส่ทุกอย่างใน prompt เดียว ให้เริ่มจาก draft แล้ว iterate — แต่ละรอบ refine ทีละ concern

📝

Step 1: Draft

"ช่วยเขียน REST API สำหรับ user authentication"

🔧

Step 2: Refine

"ดีมาก ตอนนี้เพิ่ม rate limiting และ JWT refresh token"

✨

Step 3: Polish

"เพิ่ม error handling ให้ครบ และ JSDoc comments"

❌ Mega Prompt
เขียน REST API ที่มี authentication, authorization, rate limiting, JWT, refresh token, error handling, logging, validation, unit tests, JSDoc, และ OpenAPI spec ทั้งหมดในครั้งเดียว # ใช้ token สูงมาก # ผลลัพธ์ไม่ครบทุกส่วน # แก้ยากถ้ามีส่วนผิด
✅ Iterative
# รอบ 1: Core structure เขียน Express router สำหรับ auth API # รอบ 2: Security เพิ่ม JWT + rate limiting # รอบ 3: Robustness เพิ่ม error handling + validation # รอบ 4: Docs เพิ่ม JSDoc + OpenAPI spec
🏆 Iterative = ประหยัด Token: การ re-prompt ใหม่ทั้งหมดทุกครั้งเปลือง token และ context การ iterate จาก draft ที่มีอยู่แล้วประหยัดกว่า 30-40%

Context Loading

โหลด Context อย่างมีประสิทธิภาพ — บอก file ตรงๆ

Intermediate ประหยัด Token 30-50%

แทนที่จะปล่อยให้ Claude สำรวจ codebase เอง ให้บอก file ที่ต้องอ่านตรงๆ — เร็วกว่า แม่นกว่า ประหยัดกว่า

📁

ระบุ Files

บอกชื่อ file ที่เกี่ยวข้องโดยตรง ไม่ต้องให้ Claude ค้นหาเอง

🗺️

ระบุ Context

อธิบาย architecture สั้นๆ ก่อน เช่น "ใช้ repository pattern + DI"

⚡

ใช้ .claudeignore

ตัด files ที่ไม่เกี่ยวออก node_modules, dist, logs ไม่ต้องให้ Claude อ่าน

❌ ปล่อยให้ Claude สำรวจ
ช่วย fix bug ใน authentication # Claude ต้องสำรวจ codebase # อ่าน file ที่ไม่เกี่ยวหลาย file # เสีย token สูง # อาจเข้าใจ context ผิด
✅ ระบุ Context ชัดเจน
อ่าน files เหล่านี้ก่อน: - src/auth/middleware.ts (JWT verify) - src/auth/service.ts (business logic) - src/types/user.ts (User interface) - src/config/auth.config.ts (settings) Context: ใช้ Express + Passport.js Bug: JWT token ไม่ expire ตาม config ช่วย fix และ explain root cause
ℹ️ วิธีระบุ files ให้ Claude อ่าน: พิมพ์ path ตรงๆ ใน chat — Claude จะ read file นั้นก่อน หรือใส่ file path ใน CLAUDE.md เพื่อให้ Claude รู้ว่า architecture ใช้ files ไหนเป็นหลัก

Role Assignment

ให้ Claude รับบทเป็น Expert ในด้านที่ต้องการ

Beginner High Impact

การบอกให้ Claude รับ "บทบาท" ทำให้คำตอบมี depth และ context ที่ตรงกับงานมากขึ้น — เหมือนจ้าง specialist แทน generalist

🏗️

Template

คุณคือ [role] ที่เชี่ยวชาญ [domain]
ช่วยฉัน [task]

🎯

เหมาะกับ

Code review, Architecture design, Security audit, Performance tuning

📍

วาง Role ได้ที่

ใส่ใน CLAUDE.md (ถาวร) หรือพิมพ์ในแต่ละ session

❌ ไม่มี Role
# prompt ทั่วไป ช่วยดู code นี้ให้หน่อย # ผลลัพธ์: คำตอบผิวเผิน # ไม่ได้ focus ประเด็นสำคัญ
✅ มี Role
# prompt พร้อม role คุณคือ senior TypeScript engineer ที่เชี่ยวชาญ Node.js และ security ช่วย review code นี้ โดยเน้น: - SQL injection risks - Input validation - Error handling patterns # ผลลัพธ์: ลึก ครบ ตรงจุด
🏆 Pro Tip: วาง role ใน CLAUDE.md ไม่ต้องพิมพ์ซ้ำทุก session
# CLAUDE.md ## Role คุณคือ senior full-stack engineer ที่เชี่ยวชาญ React, TypeScript, Node.js และ PostgreSQL เน้น clean code, SOLID principles, และ security best practices
🔗

Chain of Thought

ให้ Claude คิดก่อนทำ — วิเคราะห์ทีละขั้น

Intermediate Complex Tasks

แทนที่จะให้ Claude ตอบทันที บอกให้ "คิดก่อน" — ผลลัพธ์จะถูกต้องมากขึ้น โดยเฉพาะงานซับซ้อน

🧠

เหมาะกับ

Complex architecture, Debugging tricky bugs, Refactoring decisions, Security analysis

⚡

คู่กับ Plan Mode

ใช้ Shift+Tab เปิด Plan Mode + Chain of Thought = ผลลัพธ์ดีที่สุด

🔑

คำสำคัญ

"วิเคราะห์ก่อน", "คิดทีละขั้น", "ข้อดี-ข้อเสีย", "เหตุผล"

❌ ไม่ใช้ CoT
migrate database schema จาก MongoDB ไป PostgreSQL ให้หน่อย # Claude เดาแล้วลุย # อาจพลาด edge cases สำคัญ
✅ ใช้ Chain of Thought
ก่อน migrate ขอให้: 1. วิเคราะห์ schema ปัจจุบัน 2. ระบุ data types ที่ต้องแปลง 3. หา relationships ที่ซับซ้อน 4. ชี้ risks และ data loss risks 5. เสนอ migration strategy แล้วค่อย implement migration script
💡 Trigger phrases สำหรับ Chain of Thought:
"คิดทีละขั้น", "วิเคราะห์ก่อน implement", "อธิบาย reasoning", "pros and cons ก่อน", "step by step"
🎯

Few-shot Prompting

ให้ตัวอย่าง 2-3 อัน แล้วค่อยถามงานจริง

Intermediate Style Matching

Claude เรียนรู้ pattern จากตัวอย่างที่คุณให้ได้ในทันที ใช้เพื่อบังคับ code style, naming convention, หรือ output format ที่ต้องการ

🧪

เหมาะกับ

เขียน tests, สร้าง boilerplate, naming conventions, commit message style

📏

จำนวนตัวอย่าง

2-3 ตัวอย่างเพียงพอ — มากเกินไปเปลือง token ไม่ได้ผลดีขึ้น

🎨

ผลลัพธ์

Output ที่ match style โปรเจคปัจจุบัน ไม่ต้อง edit เพิ่ม

❌ ไม่มีตัวอย่าง
เขียน unit test สำหรับ UserService.createUser()
✅ Few-shot
# Test patterns ในโปรเจคนี้: describe('OrderService', () => { it('should throw if stock < 0', async () => { await expect(svc.create({qty:-1})) .rejects.toThrow('INVALID_STOCK') }) }) # ตอนนี้เขียน test สำหรับ # UserService.createUser() ในรูปแบบเดียวกัน
ℹ️ Few-shot ใน CLAUDE.md: วาง example patterns ใน CLAUDE.md ได้เลย จะ apply กับทุก session โดยอัตโนมัติ
📐

Constraint Prompting

ระบุขอบเขตชัดเจน — ห้ามทำ / ต้องทำ / ขนาดที่ต้องการ

Beginner ลด Error 40%+

Prompt ที่ไม่มีขอบเขตทำให้ Claude ตัดสินใจเองในสิ่งที่อาจไม่ต้องการ การระบุ constraint ชัดเจนช่วยลด unexpected changes

🚫

ห้ามทำ

ห้ามเปลี่ยน interface, ห้าม refactor ส่วนอื่น, ห้ามเพิ่ม dependency ใหม่

✅

ต้องทำ

ต้องผ่าน existing tests, ต้องใช้ TypeScript strict mode, ต้อง backward compatible

📏

ขนาด / Scope

แก้เฉพาะ function นี้, ไม่เกิน 50 lines, อธิบายสั้นไม่เกิน 3 bullets

❌ ไม่มี Constraint
ช่วย optimize ฟังก์ชัน getUserById() # Claude อาจ: # - เปลี่ยน function signature # - Refactor ทั้ง file # - เพิ่ม library ใหม่
✅ มี Constraint
ช่วย optimize getUserById(): ✅ แก้เฉพาะ function นี้เท่านั้น ✅ ต้องผ่าน existing unit tests ✅ ต้อง type-safe (TypeScript strict) 🚫 ห้ามเปลี่ยน function signature 🚫 ห้ามเพิ่ม external dependencies # ผลลัพธ์: targeted, safe, predictable
⚠️ Claude ใจดีเกินไป: Claude มักพยายาม "ช่วยเพิ่ม" โดยไม่ได้ถูกขอ — ระบุ constraint ป้องกันการแตะ code ที่ไม่ต้องการ
🔄

Iterative Prompting

พัฒนาทีละรอบ — draft แล้วค่อยๆ refine

Intermediate ประหยัด Token

แทนที่จะใส่ทุกอย่างใน prompt เดียว ให้เริ่มจาก draft แล้ว iterate — แต่ละรอบ refine ทีละ concern

📝

Step 1: Draft

"ช่วยเขียน REST API สำหรับ user authentication"

🔧

Step 2: Refine

"ดีมาก ตอนนี้เพิ่ม rate limiting และ JWT refresh token"

✨

Step 3: Polish

"เพิ่ม error handling ให้ครบ และ JSDoc comments"

❌ Mega Prompt
เขียน REST API ที่มี authentication, authorization, rate limiting, JWT, refresh token, error handling, logging, validation, unit tests, JSDoc, และ OpenAPI spec ทั้งหมดในครั้งเดียว # ใช้ token สูงมาก # ผลลัพธ์ไม่ครบทุกส่วน # แก้ยากถ้ามีส่วนผิด
✅ Iterative
# รอบ 1: Core structure เขียน Express router สำหรับ auth API # รอบ 2: Security เพิ่ม JWT + rate limiting # รอบ 3: Robustness เพิ่ม error handling + validation # รอบ 4: Docs เพิ่ม JSDoc + OpenAPI spec
🏆 Iterative = ประหยัด Token: การ re-prompt ใหม่ทั้งหมดทุกครั้งเปลือง token และ context การ iterate จาก draft ที่มีอยู่แล้วประหยัดกว่า 30-40%
🏗️

Context Loading

โหลด Context อย่างมีประสิทธิภาพ — บอก file ตรงๆ

Intermediate ประหยัด Token 30-50%

แทนที่จะปล่อยให้ Claude สำรวจ codebase เอง ให้บอก file ที่ต้องอ่านตรงๆ — เร็วกว่า แม่นกว่า ประหยัดกว่า

📁

ระบุ Files

บอกชื่อ file ที่เกี่ยวข้องโดยตรง ไม่ต้องให้ Claude ค้นหาเอง

🗺️

ระบุ Context

อธิบาย architecture สั้นๆ ก่อน เช่น "ใช้ repository pattern + DI"

⚡

ใช้ .claudeignore

ตัด files ที่ไม่เกี่ยวออก node_modules, dist, logs ไม่ต้องให้ Claude อ่าน

❌ ปล่อยให้ Claude สำรวจ
ช่วย fix bug ใน authentication # Claude ต้องสำรวจ codebase # อ่าน file ที่ไม่เกี่ยวหลาย file # เสีย token สูง # อาจเข้าใจ context ผิด
✅ ระบุ Context ชัดเจน
อ่าน files เหล่านี้ก่อน: - src/auth/middleware.ts (JWT verify) - src/auth/service.ts (business logic) - src/types/user.ts (User interface) - src/config/auth.config.ts (settings) Context: ใช้ Express + Passport.js Bug: JWT token ไม่ expire ตาม config ช่วย fix และ explain root cause
ℹ️ วิธีระบุ files ให้ Claude อ่าน: พิมพ์ path ตรงๆ ใน chat — Claude จะ read file นั้นก่อน หรือใส่ file path ใน CLAUDE.md เพื่อให้ Claude รู้ว่า architecture ใช้ files ไหนเป็นหลัก
📊

สรุป: เทคนิคไหนใช้เมื่อไหร่

เลือกให้ถูก situation → ผลลัพธ์ดีที่สุด

เทคนิค เหมาะกับ ผลลัพธ์
🎭 Role Assignment Code review, Architecture, Security audit คำตอบ expert-level มี depth มากขึ้น
🔗 Chain of Thought Complex bugs, Major refactoring, Architecture decisions ลด errors จากการ "ลุย" ไม่คิด
🎯 Few-shot Unit tests, Naming conventions, Commit messages Output ตรง style โปรเจคทันที
📐 Constraint Hotfixes, Targeted patches, Production code ลด unexpected changes 40%+
🔄 Iterative Feature development, API design, Large tasks ประหยัด token, คุมทิศทางได้ดีกว่า
🏗️ Context Loading Bug fix, Feature add, Code understand ประหยัด token 30-50%, แม่นขึ้น
🏆 Combo ที่ดีที่สุด: Role Assignment + Chain of Thought + Constraint Prompting = คำตอบ expert-level ที่ targeted และ predictable
